# Trackarr Tracker (Go)

Standalone BitTorrent tracker speaking **two protocols**:

- **HTTP / BEP 3** on `8080/tcp` — the classic announce / scrape over
  bencode.
- **UDP / BEP 15** on `6969/udp` — the binary, connection-id-handshake
  variant favoured by every modern client. The passkey is carried via
  the **BEP 41 URL_DATA** option, so a `.torrent` advertising
  `udp://host:6969/announce/PASSKEY` works without any client config.

Built in Go, ships as a ~13 MB `scratch`-based Docker image, and uses
[sqlc](https://sqlc.dev/) for type-safe database access.

The wire-agnostic announce processor (`Server.ProcessAnnounce`) lives in
`internal/server/handler.go` and is reused by both transports — there's
exactly one place where the passkey is resolved, the ratio gate runs,
the dedup key is stamped, deltas + bonus multipliers are applied, and
HnR / seed-time bookkeeping happens.

For the protocol details, BEP 41 passkey scheme, and operator caveats
see the [UDP Tracker guide](../../doc/guide/udp-tracker.md).

## Layout

```
apps/tracker/
├── cmd/tracker/main.go         entrypoint — boots the HTTP server and (when
│                               TRACKER_UDP_ENABLED) the UDP server
├── db/
│   ├── schema.sql              subset of the Drizzle schema (sqlc input)
│   └── queries/                hand-written SQL (sqlc input)
│       ├── users.sql
│       ├── torrents.sql
│       ├── settings.sql
│       └── hnr.sql
├── internal/
│   ├── announce/               BEP 3 query parser (HTTP-side)
│   ├── bencode/                encode-only bencode (no external lib)
│   ├── bonus/                  resolver for active bonus-event multipliers
│   ├── config/                 env loading (HTTP/UDP ports, secrets, debug)
│   ├── cryptohash/             hashIP — bit-compatible with apps/api
│   ├── db/                     pool helper + settings cache + business helpers
│   ├── peers/                  Redis store (peer JSON + key prefix)
│   ├── queries/                ⚠ generated by sqlc — do not hand-edit
│   ├── server/                 HTTP handlers + Server.ProcessAnnounce
│   └── udp/                    BEP 15 listener, BEP 41 passkey extraction,
│                               stateless connection-id HMAC
├── scripts/
│   ├── gen.sh                  regenerate internal/queries/ from db/queries/
│   └── dump-schema.sh          refresh db/schema.sql from a live Postgres
├── tests/
│   └── integration.sh          end-to-end test against real DB+Redis
├── sqlc.yaml
└── Dockerfile
```

## Running locally

```bash
DATABASE_URL=postgres://tracker:tracker@localhost:5432/trackarr \
REDIS_URL=redis://localhost:6379 \
REDIS_PASSWORD=... \
IP_HASH_SECRET=$(openssl rand -hex 32) \
go run ./cmd/tracker
```

By default the binary listens on `:8080` (HTTP) and `:6969/udp`
simultaneously. Override with `TRACKER_HTTP_PORT` / `TRACKER_UDP_PORT`,
or set `TRACKER_UDP_ENABLED=false` to keep the UDP listener off.

`IP_HASH_SECRET` keys both the per-peer IP fingerprint and the UDP
connection-id HMAC — one secret, two consumers. Test announces:

```bash
# HTTP
curl 'http://localhost:8080/announce?info_hash=…&peer_id=…&port=51413&passkey=…&left=0&compact=1'

# UDP — easiest path: feed a .torrent that advertises
# udp://host:6969/announce/PASSKEY into qBittorrent / Transmission /
# Deluge. Bare URL probing without a client is fiddly because the
# 16-byte connect handshake has to land first.
```

## Adding a new query

1. Edit `db/queries/*.sql` and add the annotated query:
   ```sql
   -- name: GetTorrentSize :one
   SELECT size FROM torrents WHERE info_hash = $1;
   ```

2. Regenerate the typed Go bindings:
   ```bash
   ./apps/tracker/scripts/gen.sh
   ```

3. Use it in your code:
   ```go
   size, err := s.db.Q.GetTorrentSize(ctx, infoHashHex)
   ```

If sqlc complains your query references an unknown column, refresh
`db/schema.sql` first:
```bash
./apps/tracker/scripts/dump-schema.sh
```

## Schema ownership

Migrations are owned by Drizzle in `packages/db/src/schema.ts`. The tracker
only keeps a **snapshot** of the columns it uses (`db/schema.sql`) so sqlc
can typecheck queries.

When the api-side schema changes:

1. Apply the Drizzle migration as usual.
2. Run `./scripts/dump-schema.sh` to refresh the tracker's snapshot.
3. Update any tracker query that touches the changed columns.
4. `./scripts/gen.sh` to regenerate.
5. The Go compiler will point you at every call site that needs an update.

## Tests

```bash
# Unit — covers the announce parser, bencode encoder, IP hash,
# UDP wire format (BEP 15 + BEP 41), and the connection-id HMAC.
go test ./...

# UDP-only — fast, no DB/Redis needed
go test ./internal/udp/...

# Integration (real Postgres + Redis via docker-compose)
./tests/integration.sh
```

The `internal/udp/` package ships **17 unit tests** including BEP 41
URL_DATA fragmentation handling, anti-spoofing checks, and the
IPv4-in-16 stability invariant (a peer that announces from `1.2.3.4`
and again from `::ffff:1.2.3.4` must keep the same connection id).
